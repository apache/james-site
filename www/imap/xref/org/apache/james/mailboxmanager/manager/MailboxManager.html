32  /**
33   * <p>
34   * Central MailboxManager which creates, lists, provides, renames and deletes
35   * Mailboxes
36   * </p>
37   * <p>
38   * An important goal is to be JavaMail feature compatible. That means JavaMail
39   * could be used in both directions: As a backend for e.g. accessing a Maildir
40   * JavaMail store or as a frontend to access a JDBC MailboxManager through
41   * JavaMail. This should be possible by not too complicated wrapper classes. Due
42   * to the complexity of JavaMail it might be impossible to avoid some
43   * limitations.
44   * </p>
45   * <p>
46   * Internally MailboxManager deals with named repositories that could have
47   * different implementations. E.g. JDBC connections to different hosts or
48   * Maildir / Mbox like stores. This repositories are identified by its names and
49   * maybe are configured in config.xml. The names of the mailboxes have to be
50   * mapped to the corresponding repository name. For user mailboxes this could be
51   * done by a "User.getRepositoryName()" property. It is imaginable that
52   * repositories lookup further properties from the user object like a path name
53   * for a file based storage method. Until Milestone 6 there is only one named
54   * repository: "default".
55   * </p>
56   * <p>
57   * The only operation that requires dealing with the named repositories directly
58   * is the quota management. It is probably really difficult to implement a quota
59   * system that spans multiple repository implementations. That is why quotas are
60   * created for a specific repository. To be able to administer, repositories and
61   * theier belonging mailboxes can be listet.
62   * </p>
63   */
